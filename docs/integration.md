## 1. Общая информация

### 1.1 Цель отчета

Провести полный анализ текущей системы FoodMiniApp и определить требования для интеграции с:

- **iiko** — система управления рестораном (меню, заказы, складской учет)
- **PremiumBonus** — система лояльности и управления клиентской базой

### 1.2 Принцип работы системы

Система должна работать в **двух режимах**:

- **Автономный режим** — без интеграций, используя собственную БД (полный доступ к редактированию)
- **Режим с интеграцией** — синхронизация с iiko (меню, заказы) и PremiumBonus (клиенты, бонусы)

**Важные особенности режима с интеграцией:**

1. **При включении iiko**:
   - Локальные модули городов, филиалов, полигонов и карты продолжают работать в FoodMiniApp
   - Из iiko синхронизируются только меню и стоп-лист; заказы отправляются в iiko
   - Редактирование меню выполняется в iiko, в FoodMiniApp меню работает как read-only

2. **При включении PremiumBonus**:
   - Новые регистрации через Telegram → автоматически создается карточка в PremiumBonus
   - Если клиент уже зарегистрирован в PremiumBonus:
     - Проверяем по номеру телефона при первом входе в Telegram Mini App
     - Загружаем существующую карточку из PremiumBonus
     - **НЕ начисляем приветственные бонусы** (клиент уже получил их ранее)
   - Локальный модуль лояльности становится неактивным, все операции идут через PremiumBonus API

3. **Локальная БД как буфер синхронизации (критически важно!)**:
   - **Все данные всегда сохраняются в локальной БД**, независимо от доступности внешних систем
   - При недоступности iiko или PremiumBonus:
     - Данные сохраняются локально со статусом `sync_status: 'pending'`
     - Система продолжает работать без сбоев
     - Накапливается очередь несинхронизированных данных
   - При восстановлении связи:
     - Автоматически запускается синхронизация накопленных изменений
     - Данные отправляются в порядке очереди
     - Статус меняется на `sync_status: 'synced'` после успешной отправки
   - Это обеспечивает **бесперебойную работу** системы и **отсутствие потери данных**

## 3. Анализ интеграции с iiko

### 3.1 Основные функции iiko API

iiko предоставляет REST API для:

- **Меню**: получение номенклатуры, категорий, модификаторов, цен
- **Заказы**: создание заказов, обновление статусов, отмена
- **Складской учет**: остатки товаров, стоп-лист
- **Webhooks**: уведомления о смене статуса заказа

### 3.1.1 Локальная адресация и доставка

В проекте используется локальная схема адресов и доставки:

- города, филиалы, полигоны и тарифы управляются только в FoodMiniApp;
- адресный справочник и геокодирование работают на стороне FoodMiniApp;
- iiko не используется как источник городов, филиалов, полигонов и справочника улиц.

### 3.2 Критические аспекты интеграции с iiko

#### 3.2.1 Синхронизация меню

**Проблема:**  
Текущая система имеет собственную структуру меню с вариантами и модификаторами, которая может не совпадать с номенклатурой iiko.

**Решение:**

**Автоматическая синхронизация меню из iiko:**

1. **Кнопка "Синхронизировать меню из iiko"** в админ-панели
   - Запрашивает полную номенклатуру из iiko API
   - Автоматически создает/обновляет категории, позиции, варианты, модификаторы в локальной БД
   - Сохраняет iiko ID в специальном поле для последующих обновлений
   - Загружает изображения из iiko (URL или файлы)

2. **Структура данных с iiko ID:**

   ```sql
   ALTER TABLE menu_items
   ADD COLUMN iiko_item_id VARCHAR(255) NULL,
   ADD COLUMN iiko_synced_at TIMESTAMP NULL,
   ADD INDEX idx_iiko_item_id (iiko_item_id);

   ALTER TABLE item_variants
   ADD COLUMN iiko_variant_id VARCHAR(255) NULL,
   ADD INDEX idx_iiko_variant_id (iiko_variant_id);

   ALTER TABLE modifiers
   ADD COLUMN iiko_modifier_id VARCHAR(255) NULL,
   ADD INDEX idx_iiko_modifier_id (iiko_modifier_id);

   ALTER TABLE menu_categories
   ADD COLUMN iiko_category_id VARCHAR(255) NULL,
   ADD INDEX idx_iiko_category_id (iiko_category_id);
   ```

3. **Процесс синхронизации:**
   - Получить полный список номенклатуры из iiko: `GET /api/1/nomenclature`
   - Для каждой категории/позиции/модификатора проверить наличие в локальной БД по `iiko_*_id`
   - Если записи нет → создать новую с сохранением `iiko_*_id`
   - Если запись есть → обновить (название, описание, цену, изображение, доступность)
   - Загрузить изображения из iiko и сохранить локально или использовать URL из iiko

4. **Периодическая автоматическая синхронизация:**
   - Воркер синхронизирует меню каждые 30 минут
   - Обновляет цены, описания, доступность
   - Логирует все изменения

5. **Что синхронизируется из iiko:**
   - ✅ Категории (названия, порядок)
   - ✅ Позиции меню (название, описание, состав, КБЖУ, вес)
   - ✅ Варианты (размеры с разными ценами)
   - ✅ Модификаторы (дополнения, соусы)
   - ✅ Цены по всем позициям
   - ✅ Изображения (URL или файлы)
   - ✅ Доступность (активность позиций)
   - ✅ Стоп-лист (позиции с нулевым остатком)

**Режим работы:**

- **С интеграцией iiko**:
  - При первом включении интеграции → кнопка "Загрузить меню из iiko"
  - Меню автоматически синхронизируется каждые 30 минут
  - Локальная БД служит кешем для быстрого доступа
  - Админ-панель FoodMiniApp переходит в режим **"только чтения"**
  - Все изменения меню возможны только в системе iiko
  - Кнопка "Обновить меню сейчас" для ручной синхронизации
- **Без интеграции**:
  - Меню полностью управляется через админ-панель FoodMiniApp
  - Доступно создание, редактирование и удаление категорий, позиций, модификаторов

#### 3.2.2 Синхронизация заказов с буферизацией

**Проблема:**  
Заказы создаются в FoodMiniApp, но должны передаваться в iiko для обработки на кухне и в кассе.

**Решение с механизмом буферизации:**

1. **Локальное сохранение (приоритет №1)**
   - Заказ ВСЕГДА сначала сохраняется в таблицу `orders`
   - Пользователь получает подтверждение создания заказа немедленно
   - Даже если iiko недоступна, заказ создан и видим в системе

2. **Асинхронная синхронизация**
   - После сохранения заказа создается задача в очереди BullMQ
   - Задача пытается отправить заказ в iiko API
   - При успехе: сохраняется `iiko_order_id`, статус → `synced`
   - При ошибке: статус → `pending` или `error`, задача переносится на повтор

3. **Автоматический retry при восстановлении связи**
   - Worker каждые 5 минут находит заказы со статусом `pending`/`error`
   - Повторно пытается отправить их в iiko
   - Максимум 5 попыток, затем статус → `failed` (требуется ручное вмешательство)

- Добавить поля в таблицу `orders`:

  ```sql
  ALTER TABLE orders
  ADD COLUMN iiko_order_id VARCHAR(255) NULL COMMENT 'ID заказа в iiko',
  ADD COLUMN iiko_sync_status ENUM('pending', 'synced', 'error', 'failed') DEFAULT 'pending',
  ADD COLUMN iiko_sync_error TEXT NULL,
  ADD COLUMN iiko_sync_attempts INT DEFAULT 0,
  ADD COLUMN iiko_last_sync_at TIMESTAMP NULL,
  ADD INDEX idx_iiko_sync_status (iiko_sync_status);
  ```

- Реализовать фоновую очередь для отправки заказов в iiko (BullMQ)
- Endpoint для webhook от iiko: `POST /api/webhooks/iiko/order-status`
- Синхронизация статусов: iiko → FoodMiniApp

**Замаппировать статусы:**
| FoodMiniApp | iiko |
|----------------|--------------------|
| pending | Unconfirmed |
| confirmed | WaitCooking |
| preparing | CookingStarted |
| ready | CookingCompleted |
| delivering | OnWay |
| completed | Delivered/Closed |
| cancelled | Cancelled |

**Важно:** При недоступности iiko кухня работает по заказам из локальной БД. После восстановления связи все заказы автоматически синхронизируются.

#### 3.2.3 Стоп-лист и остатки

**Проблема:**  
Стоп-лист в FoodMiniApp управляется вручную, в iiko он зависит от остатков на складе.

**Решение:**

- Периодически запрашивать остатки из iiko (каждые 5 минут)
- Автоматически добавлять в стоп-лист позиции с нулевым остатком
- Учитывать полуфабрикаты и расчет остатков на основе рецептуры

#### 3.2.4 Модификаторы

**Проблема:**  
Модификаторы в FoodMiniApp имеют гибкую структуру (группы, обязательные/необязательные), в iiko они привязаны к блюдам через "модификаторные схемы".

**Решение:**

- При автоматической синхронизации меню из iiko модификаторы загружаются вместе с позициями
- Система автоматически создает группы модификаторов и отдельные модификаторы
- В локальной БД сохраняется `iiko_modifier_id` для последующих обновлений
- Цены модификаторов обновляются при каждой синхронизации
- Привязка модификаторов к позициям осуществляется автоматически на основе данных из iiko

**Структура синхронизации:**

```
iiko Модификаторы → FoodMiniApp
├── Группа модификаторов (создается автоматически)
│   ├── Название
│   ├── Тип (single/multiple)
│   ├── Обязательность
│   └── iiko_modifier_group_id
├── Модификаторы в группе
│   ├── Название
│   ├── Цена
│   ├── Изображение (если есть)
│   └── iiko_modifier_id
└── Связь с позицией меню (автоматическая)
```

## 4. Анализ интеграции с PremiumBonus

### 4.1 Основные функции PremiumBonus API

PremiumBonus предоставляет REST API для:

- **Клиенты**: регистрация, получение информации, редактирование
- **Бонусы**: начисление, списание, баланс, история
- **Покупки**: создание покупки, эмуляция (dry-run), отмена
- **Группы клиентов**: уровни лояльности
- **Промокоды**: активация, проверка
- **Коды подтверждения**: отправка SMS, проверка

### 4.2 Критические аспекты интеграции с PremiumBonus

#### 4.2.1 Замена модуля лояльности

**Проблема:**  
Текущая система имеет собственную реализацию лояльности (таблицы `loyalty_levels`, `user_loyalty_levels`, `loyalty_transactions`), которая **должна быть полностью заменена** на PremiumBonus.

**Решение:**

- **Сохранить таблицы** для режима без интеграции
- Добавить поле в `users`:

  ```sql
  ALTER TABLE users
  ADD COLUMN pb_client_id VARCHAR(255) NULL COMMENT 'ID клиента в PremiumBonus',
  ADD COLUMN pb_external_id VARCHAR(255) NULL COMMENT 'external_id для PremiumBonus',
  ADD COLUMN loyalty_mode ENUM('local', 'premiumbonus') DEFAULT 'local',
  ADD INDEX idx_pb_client_id (pb_client_id);
  ```

- Создать **адаптер лояльности** (`loyaltyAdapter.js`), который в зависимости от режима (`loyalty_mode`) направляет запросы либо в локальную БД, либо в PremiumBonus API

- Реализовать методы:
  - `getUserBalance(userId)` — получение баланса
  - `earnBonuses(userId, amount, reason)` — начисление
  - `spendBonuses(userId, amount, reason)` — списание
  - `getUserLevel(userId)` — текущий уровень
  - `getTransactionHistory(userId)` — история

**Режим работы:**
| Режим | Источник данных |
|-------------------|------------------------|
| `local` | Локальная БД |
| `premiumbonus` | PremiumBonus API |

#### 4.2.2 Синхронизация клиентов с буферизацией

**Проблема:**  
Клиенты регистрируются в FoodMiniApp через Telegram, но должны быть зарегистрированы в PremiumBonus для работы с бонусами.

**Механизм буферизации для клиентов:**

1. **Локальное сохранение (приоритет №1)**
   - Клиент ВСЕГДА сначала регистрируется в локальной таблице `users`
   - Пользователь получает доступ к приложению немедленно
   - Даже если PremiumBonus недоступен, клиент работает с системой

2. **Асинхронная синхронизация с PremiumBonus**
   - После регистрации создается задача в очереди для проверки/создания в PB
   - При успехе: сохраняется `pb_client_id`, статус → `synced`
   - При ошибке: статус → `pending`, задача переносится на повтор

3. **Добавить поля для синхронизации:**

```sql
ALTER TABLE users
ADD COLUMN pb_client_id VARCHAR(255) NULL COMMENT 'ID клиента в PremiumBonus',
ADD COLUMN pb_sync_status ENUM('pending', 'synced', 'error', 'failed') DEFAULT 'pending',
ADD COLUMN pb_sync_error TEXT NULL,
ADD COLUMN pb_sync_attempts INT DEFAULT 0,
ADD COLUMN pb_last_sync_at TIMESTAMP NULL,
ADD INDEX idx_pb_sync_status (pb_sync_status);
```

**Решение:**

**Сценарий 1: Новый клиент (нет в PremiumBonus)**

1. Пользователь первый раз авторизуется в Telegram Mini App и предоставляет номер телефона
2. Создать профиль в локальной БД FoodMiniApp (статус `pb_sync_status: 'pending'`)
3. Запустить фоновую задачу:
   - Проверить наличие клиента в PremiumBonus: `POST /buyer-info` с номером телефона
   - Если клиента нет (ответ `is_registered: false`):
     - Создать профиль в PremiumBonus через `POST /buyer-register`
     - Передать: `phone`, `first_name`, `last_name`, `email`, `birth_date`, `external_id`
     - Сохранить `client_id` из ответа в `users.pb_client_id`
     - **Начислить приветственные бонусы** (настраивается в PremiumBonus)
     - Обновить статус → `synced`
   - При ошибке: статус → `error`, повтор через 5 минут

**Сценарий 2: Существующий клиент PremiumBonus (первый вход в Telegram Mini App)**

1. Пользователь первый раз авторизуется в Telegram Mini App и предоставляет номер телефона
2. Создать профиль в локальной БД FoodMiniApp (статус `pb_sync_status: 'pending'`)
3. Запустить фоновую задачу:
   - Проверить наличие клиента в PremiumBonus: `POST /buyer-info` с номером телефона
   - Если клиент найден (ответ `is_registered: true`):
     - Загрузить данные клиента из PremiumBonus (`client_id`, баланс, уровень, профиль)
     - Обновить локальную запись с сохранением `pb_client_id`
     - Связать Telegram ID с существующей карточкой PremiumBonus
     - **НЕ начислять приветственные бонусы** (клиент уже зарегистрирован)
     - Обновить статус → `synced`
   - При ошибке: статус → `error`, повтор через 5 минут

**Сценарий 3: Обновление профиля с буферизацией**

- При изменении данных в профиле Telegram Mini App (имя, email, дата рождения):
  1. Обновить локальную запись в FoodMiniApp
  2. Поставить задачу в очередь на синхронизацию с PremiumBonus
  3. Фоновый worker отправит изменения через `POST /buyer-edit`
  4. При ошибке: данные останутся в очереди на повтор

**Сценарий 4: Получение актуальных данных**

- Баланс и уровень лояльности запрашиваются из PremiumBonus при каждом открытии приложения
- Метод: `POST /buyer-info` с `identificator` (телефон или `client_id`)
- Кешировать в Redis на 5 минут для снижения нагрузки
- При недоступности PremiumBonus: показывать последние данные из локальной БД

**Важно:** Клиент всегда может пользоваться приложением, даже если PremiumBonus недоступен. Все операции будут синхронизированы автоматически после восстановления связи.

#### 4.2.3 Начисление и списание бонусов с буферизацией

**Проблема:**  
Текущая система начисляет бонусы при завершении заказа, списывает при оформлении. PremiumBonus требует создания "покупки" с детализацией состава заказа.

**Механизм буферизации для покупок:**

1. **Локальное сохранение заказа (приоритет №1)**
   - Заказ ВСЕГДА создается в локальной БД с информацией об использованных бонусах
   - Пользователь видит заказ в системе немедленно
   - Даже если PremiumBonus недоступен, заказ принят

2. **Асинхронная синхронизация покупки**
   - После создания заказа задача в очереди отправляет `POST /purchase` в PB
   - При успехе: сохраняется `pb_purchase_id`, статус → `synced`
   - При ошибке: статус → `pending`, повтор через 5 минут

3. **Обновление статуса покупки**
   - При завершении/отмене заказа задача отправляет соответствующий статус в PB
   - При недоступности PB: статус остается `pending`, синхронизация при восстановлении

**Решение:**

**При оформлении заказа:**

1. Асинхронно вызвать `POST /purchase-request` для расчета максимальной доступной выгоды
   - При успехе: отобразить клиенту доступные бонусы
   - При ошибке: показать последние данные баланса из локальной БД
2. Создать заказ в локальной БД с полем `bonuses_used` (статус `pb_sync_status: 'pending'`)
3. Поставить в очередь задачу на создание покупки в PB: `POST /purchase`
4. Фоновый worker отправит покупку и сохранит `pb_purchase_id`

**При завершении заказа:**

1. Обновить статус заказа в локальной БД
2. Поставить в очередь задачу на изменение статуса в PB: `POST /change-purchase-status`
3. Фоновый worker отправит новый статус, получит начисленные бонусы

**При отмене заказа:**

1. Обновить статус заказа в локальной БД
2. Поставить в очередь задачу на отмену в PB: `POST /cancel-purchase`
3. Фоновый worker отменит покупку, бонусы будут возвращены

**Добавить поля в таблицу `orders`:**

```sql
ALTER TABLE orders
ADD COLUMN pb_purchase_id VARCHAR(255) NULL COMMENT 'ID покупки в PremiumBonus',
ADD COLUMN pb_sync_status ENUM('pending', 'synced', 'error', 'failed') DEFAULT 'pending',
ADD COLUMN pb_sync_error TEXT NULL,
ADD COLUMN pb_sync_attempts INT DEFAULT 0,
ADD COLUMN pb_last_sync_at TIMESTAMP NULL,
ADD INDEX idx_pb_purchase_id (pb_purchase_id),
ADD INDEX idx_pb_sync_status (pb_sync_status);
```

**Важно:** Клиент видит списанные бонусы в заказе сразу (локально). Фактическое списание в PremiumBonus происходит асинхронно. При восстановлении связи все покупки автоматически синхронизируются.

#### 4.2.4 Уровни лояльности

**Проблема:**  
Уровни лояльности в текущей системе жестко закодированы (Бронза, Серебро, Золото), в PremiumBonus они настраиваются через админ-панель PremiumBonus.

**Решение:**

- Периодически запрашивать список групп клиентов из PremiumBonus:
  - Метод: `POST /buyer-groups`
  - Кешировать в Redis на 1 час

- При расчете начислений/списаний использовать параметры группы из PremiumBonus
- Отображать текущий уровень клиента из PremiumBonus в Mini App

#### 4.2.5 Триггерные бонусы (день рождения, регистрация)

**Проблема:**  
В текущей системе триггерные бонусы начисляются через background workers. В PremiumBonus это настраивается через "сценарии".

**Решение:**

- **С интеграцией PremiumBonus**: отключить локальные воркеры, настроить сценарии в панели PremiumBonus
- **Без интеграции**: использовать локальные воркеры

**Альтернатива:** использовать метод `POST /trigger` для запуска событий из FoodMiniApp

**Статусы синхронизации:**

- `pending` — ожидает отправки (новая запись)
- `synced` — успешно синхронизировано с внешней системой
- `error` — ошибка при синхронизации (будут повторные попытки)
- `failed` — превышено максимальное количество попыток (требуется ручное вмешательство)

**Автоматическое восстановление синхронизации:**

- Periodic worker запускается каждые 5 минут
- Находит все записи со статусом `pending` или `error` (макс. 5 попыток)
- Пытается отправить их во внешние системы
- При успехе обновляет статус на `synced`

| Просмотр          | ✅ Доступно           | ✅ Доступно (read-only) |

| Синхронизация     | Не требуется          | Каждые 30 минут         |

   - Добавить поля синхронизации для заказов в `orders`:

   - `sync:iiko:menu` — автоматическая синхронизация меню из iiko (каждые 30 мин)
   - `sync:iiko:menu:manual` — ручная синхронизация по кнопке в админ-панели

   - `sync:iiko:orders:retry` — worker для повторной отправки pending/error заказов
   - `sync:premiumbonus:clients` — синхронизация клиентов (при регистрации/обновлении)
   - `sync:premiumbonus:clients:retry` — worker для повторной регистрации pending/error клиентов
   - `sync:premiumbonus:purchases` — синхронизация покупок (при создании/завершении заказа)
   - `sync:premiumbonus:purchases:retry` — worker для повторной отправки pending/error покупок

3. **Worker'ы для обработки очередей буферизации**
   - `iikoOrderSyncWorker` — каждые 5 минут обрабатывает заказы с `iiko_sync_status: 'pending'/'error'`
   - `pbClientSyncWorker` — каждые 5 минут обрабатывает клиентов с `pb_sync_status: 'pending'/'error'`
   - `pbPurchaseSyncWorker` — каждые 5 минут обрабатывает покупки с `pb_sync_status: 'pending'/'error'`
   - Макс. 5 попыток, затем статус → `failed` с уведомлением админу

4. **Кеширование внешних запросов**
   - Кешировать меню из iiko в Redis (TTL: 30 минут)
   - Полная копия меню в локальной БД (для быстрого доступа и fallback)
   - Кешировать баланс клиента из PremiumBonus (TTL: 5 минут)
   - Кешировать группы клиентов из PremiumBonus (TTL: 1 час)

  ```javascript
    if (settings.premiumbonus_enabled && ["POST", "PUT", "DELETE"].includes(req.method)) {
      return res.status(403).json({
        error: "Loyalty management is disabled when PremiumBonus integration is active",
        message: "Please manage loyalty in PremiumBonus system",
      });
    }
    next();
  };
  ```

## 7. Функции, которые становятся недоступны при интеграции

### 7.1 При включении интеграции с iiko (меню)

**Становятся недоступны в UI админ-панели (режим "только чтения"):**

- ❌ Создание/редактирование/удаление категорий (управляется в iiko)
- ❌ Создание/редактирование/удаление позиций меню (управляется в iiko)
- ❌ Создание/редактирование/удаление модификаторов (управляется в iiko)
- ❌ Управление ценами (цены берутся из iiko)
- ❌ Ручное управление стоп-листом (управляется из iiko по остаткам)

**Остаются доступными:**

- ✅ Просмотр меню в режиме чтения (read-only)
- ✅ Маппинг позиций с iiko ID (связывание локальных ID с ID в iiko)
- ✅ Управление тегами (локальная функция, не синхронизируется с iiko)
- ✅ Управление изображениями (если iiko не предоставляет фото или нужны локальные варианты)
- ✅ Управление городами, филиалами, полигонами и картой в FoodMiniApp
- ✅ Просмотр логов синхронизации

**Техническая реализация:**

- Кнопки "Создать", "Редактировать", "Удалить" скрываются или блокируются в UI
- При попытке прямого POST/PUT/DELETE запроса к API возвращается ошибка с кодом 403:
  ```json
  {
    "error": "Menu editing is disabled. Integration with iiko is active. Please manage menu in iiko system."
  }
  ```
- Данные в локальной БД обновляются только через синхронизацию из iiko

### 7.2 При включении интеграции с PremiumBonus (лояльность)

**Становятся недоступны в UI админ-панели:**

- ❌ Управление уровнями лояльности (настраивается в панели PremiumBonus)
- ❌ Ручное начисление/списание бонусов (делается через панель PremiumBonus)
- ❌ Настройки лояльности: процент начисления, срок истечения, условия уровней (в PremiumBonus)
- ❌ Триггерные настройки (день рождения, регистрация) — настраиваются в сценариях PremiumBonus

**Остаются доступными:**

- ✅ Просмотр баланса клиента (читается из PremiumBonus)
- ✅ Просмотр истории транзакций (читается из PremiumBonus)
- ✅ Просмотр текущего уровня клиента
- ✅ Просмотр профиля клиента
- ✅ Просмотр статистики по лояльности

**Техническая реализация:**

- Разделы управления лояльностью в админ-панели скрываются при `premiumbonus_enabled: true`
- Запросы к локальным endpoint'ам лояльности перенаправляются на PremiumBonus через адаптер
- Локальные таблицы `loyalty_transactions`, `user_loyalty_levels` не используются, но сохраняются для возможного возврата к автономному режиму

### 7.3 Что НЕ удаляется

**Важно:** Локальные модули и функции **не удаляются** из кода и БД, а переводятся в неактивное состояние при включении интеграции. Это позволяет:

1. **Быстро переключаться** между режимами работы через настройки
2. **Сохранить данные** при временном отключении интеграции
3. **Иметь backup** на случай недоступности внешних сервисов
4. **Легко вернуться** к автономному режиму при необходимости

**Механизм переключения:**

- Админ-панель: "Настройки → Интеграции → Включить интеграцию с iiko/PremiumBonus"
- При включении: UI блокируется, запросы идут через адаптеры к внешним API
- При выключении: UI разблокируется, запросы идут к локальной БД

1. **Модуль регистрации клиентов**
   - Проверка существования клиента в PremiumBonus при первом входе: `POST /buyer-info`
   - Автоматическая регистрация НОВЫХ клиентов в PremiumBonus: `POST /buyer-register`
   - Загрузка данных СУЩЕСТВУЮЩИХ клиентов из PremiumBonus
   - Условная логика начисления приветственных бонусов:

     ```javascript
     // Псевдокод
     async function registerOrLinkClient(phone, userData) {
       // Проверяем, есть ли клиент в PremiumBonus
       const pbClient = await premiumBonusAPI.getBuyerInfo(phone);

       if (pbClient.is_registered) {
         // Клиент уже есть в PB - загружаем данные
         await createLocalUser({
           ...userData,
           pb_client_id: pbClient.client_id,
           loyalty_mode: "premiumbonus",
         });
         // НЕ начисляем приветственные бонусы
         return { isNew: false, client: pbClient };
       } else {
         // Новый клиент - регистрируем в PB
         const newPbClient = await premiumBonusAPI.registerBuyer({
           phone: phone,
           ...userData,
         });
         await createLocalUser({
           ...userData,
           pb_client_id: newPbClient.client_id,
           loyalty_mode: "premiumbonus",
         });
         // Приветственные бонусы начислит PremiumBonus автоматически
         return { isNew: true, client: newPbClient };
       }
     }
     ```

   - Синхронизация профиля при обновлении: `POST /buyer-edit`
   - Хранение `client_id` и `external_id` в локальной БД

2. **Модуль начисления/списания бонусов (с очередью)**
   - Интеграция с процессом оформления заказа
   - Вызов `POST /purchase-request` для расчета выгоды (асинхронно)
   - Постановка в очередь задачи на создание покупки: `POST /purchase`
   - При завершении заказа: постановка в очередь задачи `POST /change-purchase-status`
   - При отмене заказа: постановка в очередь задачи `POST /cancel-purchase`
   - При успехе: сохранение `pb_purchase_id`, статус → `synced`
   - При ошибке: статус → `pending`/`error`, повтор через 5 минут

3. **Worker для обработки очереди покупок**
   - Каждые 5 минут находит покупки со статусом `pending`/`error`
   - Пытается отправить их в PremiumBonus API
   - Обрабатывает создание, изменение статуса и отмену покупок
   - При успехе: обновляет статус на `synced`
   - При превышении лимита попыток: статус → `failed`, уведомление админу

4. **Worker для обработки очереди регистрации клиентов**

   - Регистрирует новых или связывает существующих
   - При успехе: сохраняет `pb_client_id`, статус → `synced`
   - При ошибке: повтор через 5 минут (макс. 5 попыток)

5. **Синхронизация уровней лояльности**

8. **Адаптер лояльности**

1. **Несовместимость структур данных**
   - Риск: структура меню iiko может кардинально отличаться от FoodMiniApp
   - Решение: гибкий маппинг, fallback на локальные данные

2. **Производительность внешних API**
   - Риск: медленные ответы от iiko/PremiumBonus тормозят работу системы
   - Решение: агрессивное кеширование, асинхронные очереди, таймауты

3. **Доступность API (решено через буферизацию)**
   - Риск: падение iiko/PremiumBonus приводит к остановке работы
   - Решение с использованием локальной БД как буфера:
     - **Для меню (iiko)**: использовать кеш из локальной БД (последняя успешная синхронизация)
     - **Для заказов (iiko)**: заказы всегда сохраняются локально ПЕРВЫМИ, затем ставятся в очередь на отправку в iiko. При недоступности iiko заказы накапливаются в очереди (статус `pending`), автоматически отправляются при восстановлении связи
     - **Для клиентов (PremiumBonus)**: регистрация всегда происходит локально ПЕРВОЙ, затем асинхронно в PB. При недоступности PB клиент работает с системой, синхронизация произойдет автоматически
     - **Для лояльности (PremiumBonus)**: баланс кешируется в Redis, при недоступности API показывать последние известные данные. Операции с бонусами сохраняются локально и синхронизируются асинхронно
     - **Повторные попытки**: worker каждые 5 минут обрабатывает все записи со статусом `pending`/`error`, максимум 5 попыток
     - Показывать уведомление администратору о количестве несинхронизированных записей
     - **Результат**: система работает ВСЕГДА, даже без связи с внешними системами

4. **Дублирование заказов**
   - Риск: при ошибке сети заказ может быть отправлен в iiko дважды
   - Решение: идемпотентность запросов, хранение `external_order_id`, проверка дубликатов

5. **Рассинхронизация статусов**
   - Риск: статус в FoodMiniApp и iiko не совпадают
   - Решение: webhook от iiko, периодическая сверка статусов

### 10.2 Бизнес-риски

1. **Переход от локальной лояльности к PremiumBonus**
   - Риск: потеря истории бонусов клиентов
   - Решение: миграция данных, единовременное начисление текущего баланса в PremiumBonus

2. **Изменение условий лояльности**
   - Риск: клиенты недовольны новыми условиями PremiumBonus
   - Решение: плавный переход, информирование клиентов, компенсация

3. **Зависимость от сторонних сервисов**
   - Риск: при прекращении работы с iiko/PremiumBonus придется переделывать систему
   - Решение: поддержка автономного режима, минимизация привязки к конкретным API

### 10.3 Организационные риски

1. **Сложность настройки**
   - Риск: персонал не сможет настроить интеграции
   - Решение: подробная документация, обучение, техподдержка

2. **Увеличение времени разработки**
   - Риск: интеграция займет больше времени, чем планировалось
   - Решение: поэтапное внедрение, MVP для каждого этапа

## Заключение

Интеграция с iiko и PremiumBonus — это **масштабный проект**, который требует:

- Создания новой инфраструктуры для синхронизации
- Рефакторинга существующих модулей
- Разработки адаптеров для поддержки двух режимов работы
- Тщательного тестирования

**Ключевые принципы интеграции:**

1. **Дуальный режим работы** — система может работать как полностью автономно, так и с интеграциями
2. **Режим "только чтения" для меню** — при включении iiko локальный модуль меню недоступен для редактирования, данные берутся из iiko
3. **Сохранение локальных модулей** — они НЕ удаляются, а переводятся в неактивное состояние, что позволяет быстро переключаться между режимами
4. **Умная регистрация клиентов** — система проверяет наличие клиента в PremiumBonus перед регистрацией и не дублирует приветственные бонусы
5. **Локальная БД как буфер синхронизации (КРИТИЧЕСКИ ВАЖНО!)** — ВСЕ данные (заказы, клиенты, покупки) ВСЕГДА сначала сохраняются в локальной БД, затем асинхронно синхронизируются с внешними системами через очереди. При недоступности внешних систем данные накапливаются в очереди и автоматически отправляются при восстановлении связи
6. **Автоматическое восстановление синхронизации** — worker'ы каждые 5 минут обрабатывают записи со статусом `pending`/`error`, обеспечивая нулевую потерю данных
7. **Кеширование и fallback** — при недоступности внешних API используются закешированные данные

**Главное преимущество:** система остается гибкой и может работать в любом из трех режимов:

- Полностью автономно (без интеграций)
- С интеграцией только iiko
- С интеграцией только PremiumBonus
- С обеими интеграциями одновременно

**Критически важно:** система работает ВСЕГДА, даже при полной недоступности внешних систем. Клиенты могут делать заказы, регистрироваться, использовать бонусы — все данные будут синхронизированы автоматически после восстановления связи.

**Главный вызов:** обеспечить надежность и производительность при работе с внешними API, минимизировать риски рассинхронизации данных, правильно обработать состояния "только чтения" в UI, реализовать надежную систему очередей и повторных попыток.

Рекомендуется начать с **интеграции меню из iiko** (этап 2), так как это наименее критичная часть, но даст понимание всех сложностей работы с внешними API и переключения режимов.
