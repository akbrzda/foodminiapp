# FoodMiniApp — стандарты кода, поддержки и сопровождения

## 1. Цель документа

Этот документ задает единые правила для разработки, сопровождения и чтения кода в проекте FoodMiniApp.

Цели:

- снизить стоимость поддержки;
- сделать код предсказуемым;
- убрать хаос стилей между слоями;
- упростить onboarding;
- уменьшить количество скрытых багов;
- сделать PR меньше, понятнее и безопаснее.

Документ обязателен для:

- `backend/`
- `admin-panel/`
- `telegram-miniapp/`
- `bot/`
- `.github/workflows/`
- `docs/`

## 2. Базовые принципы

### 2.1. Понятность важнее умности

Код должен быть очевидным для следующего разработчика.

Запрещено:

- писать «хитрые» конструкции ради краткости;
- смешивать несколько уровней абстракции в одном файле;
- прятать важную бизнес-логику в router, middleware или UI-компонентах.

### 2.2. Один файл — одна ответственность

Если файл отвечает сразу за маршруты, валидацию, доступы, запросы в БД и формат ответа, файл надо разделить.

### 2.3. Один слой — одна роль

- `router` знает только маршруты и middleware.
- `controller` знает HTTP-вход и HTTP-выход.
- `service` знает бизнес-логику.
- `repository` знает БД.
- `store` знает состояние UI и orchestration только в рамках UI.
- `component` отвечает за отображение.

### 2.4. Границы системы валидируются на входе

Любые входные данные валидируются на границе:

- HTTP request
- WebSocket message
- webhook payload
- env variables
- background jobs

### 2.5. Ошибки должны быть предсказуемыми

Никаких пустых `catch`.
Никаких молча проглоченных ошибок.
Никаких случайных `console.error` в одном месте и `logger.error` в другом.

### 2.6. Документация должна совпадать с реальностью

README, `.env.example`, workflow secrets, `package.json` и код должны описывать один и тот же проект.

## 3. Стандарты структуры репозитория

## 3.1. Корневая структура

Целевая структура:

```text
/
  .github/
  admin-panel/
  backend/
  bot/
  telegram-miniapp/
  docs/
  package.json
  pnpm-workspace.yaml | package-workspaces
  .editorconfig
  .eslintrc.cjs | eslint.config.js
  .prettierrc
```

### Правила

- В корне должны быть единые команды для запуска и проверки.
- Все проекты должны запускаться через workspace-уровень.
- Общие правила форматирования и lint должны жить в корне.
- Дублирование конфигов между приложениями должно быть минимальным.

## 3.2. Структура backend

Целевая структура каждого доменного модуля:

```text
backend/src/modules/auth/
  routes.js
  controller.js
  service.js
  repository.js
  validators.js
  mapper.js
  constants.js
  errors.js
```

### Правила

- SQL не пишется в `routes.js`.
- `routes.js` только регистрирует маршруты.
- Все запросы к БД лежат в `repository.js`.
- Бизнес-правила лежат в `service.js`.
- Проверка request body/query/params лежит в `validators.js`.
- Преобразование ответа лежит в `mapper.js` или `serializer.js`.

## 3.3. Структура frontend

Целевая структура для `admin-panel` и `telegram-miniapp`:

```text
src/
  app/
  router/
    index.js
    guards/
    routes/
  shared/
    api/
    stores/
    ui/
    utils/
    constants/
    validators/
  modules/
    orders/
      api/
      components/
      stores/
      views/
      utils/
```

### Правила

- `router/index.js` собирает готовые массивы маршрутов.
- Guards лежат отдельно.
- API-запросы не пишутся внутри компонентов, если они используются более чем в одном месте.
- Большие фичи должны жить в `modules/`, а не в `shared/`.
- `shared/` содержит только переиспользуемые вещи.

## 3.4. Структура bot

Целевая структура:

```text
bot/src/
  config/
  modules/
    command-bot/
      handlers/
      service.js
      validators.js
  server/
  workers/
  shared/
    logger/
    errors/
    utils/
```

## 4. Ограничения на размер и сложность

### 4.1. Размер файлов

Нормы:

- до 150 строк — нормально;
- 150–250 строк — допустимо, если файл действительно цельный;
- больше 250 строк — нужен разбор;
- больше 350 строк — файл должен быть разделен.

Исключения:

- schema files;
- generated files;
- словари констант;
- migrations.

### 4.2. Размер функций

Нормы:

- до 30 строк — хорошо;
- 30–50 строк — допустимо;
- больше 50 строк — дробить.

### 4.3. Вложенность

Максимум 3 уровня вложенности.
Если нужно больше — выносить условия в именованные функции.

### 4.4. Параметры

У функции не должно быть больше 4 аргументов.
Если больше — передавать объект.

## 5. Стандарты именования

### 5.1. Файлы

- Vue-компоненты: `PascalCase.vue`
- composables: `useCart.js`, `useAuth.js`
- stores: `authStore.js` или `useAuthStore.js`, один стиль на весь репозиторий
- utils: `formatPrice.js`, `buildOrderPayload.js`
- route files: `orders.routes.js`, `menu.routes.js`
- validators: `order.validator.js` или `order.validators.js`, один стиль на весь репозиторий

Рекомендуемый стандарт:

- компоненты: `PascalCase.vue`
- все JS-файлы: `kebab-case` или доменное имя с суффиксом
- один формат выбирается и фиксируется

### 5.2. Переменные и функции

- `camelCase` для переменных и функций
- `PascalCase` для компонентов, классов, конструкторов
- `UPPER_SNAKE_CASE` для констант верхнего уровня

### 5.3. Булевы значения

Булевые переменные должны звучать как вопрос:

- `isLoading`
- `hasAccess`
- `canEdit`
- `shouldRetry`

Запрещено:

- `flag`
- `statusOk`
- `value1`

## 6. Импорты

Порядок импортов:

1. внешние библиотеки;
2. абсолютные внутренние импорты;
3. относительные внутренние импорты;
4. стили.

Пример:

```js
import { createRouter } from "vue-router";
import { defineStore } from "pinia";

import { apiClient } from "@/shared/api/client";
import { useAuthStore } from "@/shared/stores/authStore";

import { mapOrderToView } from "../utils/mapOrderToView";

import "./styles.css";
```

Правила:

- не делать циклические зависимости;
- не импортировать модули с побочными эффектами без явной причины;
- не смешивать alias и глубокие относительные пути без нужды.

## 7. Форматирование

Обязательные правила:

- единый форматтер для всех пакетов;
- `.editorconfig` обязателен;
- конец строки: LF;
- отступ: 2 пробела;
- trailing commas — по конфигу formatter;
- строка до 100 или 110 символов, один лимит на весь проект.

## 8. Комментарии и документация

### 8.1. Комментарии

Комментарии нужны только там, где без них трудно понять причину.

Нельзя:

```js
// увеличиваем count на 1
count += 1;
```

Нужно:

```js
// Сохраняем отдельный снимок корзины, чтобы восстановить ее после неуспешной оплаты.
```

### 8.2. JSDoc

JSDoc обязателен для:

- публичных helper-функций;
- сервисов с нетривиальным контрактом;
- mapper/serializer функций;
- функций, где типы аргументов неочевидны.

Пример:

```js
/**
 * Собирает payload заказа для backend API.
 * @param {Object} cart
 * @param {Object} delivery
 * @returns {Object}
 */
function buildOrderPayload(cart, delivery) {}
```

### 8.3. Документация в `docs/`

Каждый документ должен содержать:

- цель;
- границы;
- термины;
- текущее состояние;
- целевое состояние;
- ограничения;
- список открытых вопросов.

## 9. Frontend-стандарты

## 9.1. Компоненты

Компонент не должен:

- содержать длинную бизнес-логику;
- делать больше одного основного API-сценария;
- управлять роутингом, сетью и состоянием одновременно.

Компонент должен:

- быть тонким;
- получать данные через props/store/composable;
- иметь понятный публичный интерфейс;
- иметь предсказуемые emitted events.

### Признак плохого компонента

Если компонент:

- грузит данные;
- валидирует форму;
- обрабатывает ошибки;
- считает итоговую цену;
- управляет модалкой;
- делает редиректы,

то это уже не компонент, а orchestration-узел. Его надо разделить.

## 9.2. Stores

Store — это не свалка.

Store должен:

- хранить состояние;
- предоставлять actions;
- держать derived state в getters/computed;
- координировать UI-поток, если это действительно UI-flow.

Store не должен:

- содержать прямой DOM-код;
- напрямую менять router UI-эффекты без отдельного adapter/composable;
- содержать дублирующие куски API-кода.

### Рекомендуемый формат store

- `state`
- `getters/computed`
- `actions`
- `private helpers`

## 9.3. Router

Router не должен быть god-file.

Правила:

- маршруты разбиваются по доменам;
- guards лежат отдельно;
- проверки прав — отдельно;
- lazy loading обязателен для страниц;
- router не должен содержать бизнес-логику заказа, корзины, доставки и т.д.

### Целевой пример

```text
router/
  index.js
  guards/
    authGuard.js
    permissionGuard.js
    locationGuard.js
  routes/
    auth.routes.js
    orders.routes.js
    menu.routes.js
    settings.routes.js
```

## 9.4. API слой

Нужен единый API client.

Правила:

- один клиент на приложение;
- interceptors в одном месте;
- ошибки нормализуются;
- endpoint-функции лежат рядом с доменом;
- UI не должен знать детали `axios`.

## 10. Backend-стандарты

## 10.1. Routes

`routes.js` должен содержать:

- определение маршрутов;
- middleware;
- вызов controller.

`routes.js` не должен содержать:

- SQL;
- длинную бизнес-логику;
- формирование cookie;
- расчеты;
- сложные проверки доступа.

## 10.2. Controller

Controller отвечает за:

- чтение `req`;
- вызов service;
- отправку `res`;
- перевод доменных ошибок в HTTP response.

Controller не отвечает за:

- хранение SQL;
- побочные эффекты в БД;
- сложные вычисления.

## 10.3. Service

Service отвечает за бизнес-логику:

- правила доступа;
- правила перехода статусов;
- координацию нескольких repository;
- публикацию событий;
- работу с очередями.

## 10.4. Repository

Repository отвечает только за доступ к данным.

Правила:

- запросы параметризуются;
- нет бизнес-ветвлений, не связанных с БД;
- имена методов доменные: `findUserByTelegramId`, `createOrder`, `updateOrderStatus`.

## 10.5. Валидация

Рекомендуемый стандарт:

- `zod` или `joi` на входах;
- схема рядом с модулем;
- контроллер работает только с провалидированными данными.

## 10.6. Ошибки

Нужен единый набор ошибок:

- `ValidationError`
- `UnauthorizedError`
- `ForbiddenError`
- `NotFoundError`
- `ConflictError`
- `ExternalServiceError`

Каждая ошибка должна иметь:

- `code`
- `message`
- `httpStatus`
- `details` при необходимости

## 10.7. Логирование

Нужен единый logger.

Правила:

- не использовать `console.log`, `console.error` в runtime-коде;
- логировать структурированно;
- всегда включать контекст: `requestId`, `userId`, `orderId`, `module`.

Пример:

```js
logger.error("order_create_failed", {
  requestId,
  userId,
  orderId,
  error: error.message,
});
```

## 10.8. Конфигурация

Все env-переменные должны:

- быть описаны в `.env.example`;
- валидироваться на старте;
- иметь один источник правды;
- не расходиться между README и CI/CD.

## 11. Стандарты тестирования

## 11.1. Минимальный набор

### Backend

- unit tests для сервисов;
- integration tests для repository и API;
- smoke tests для auth и critical endpoints.

### Frontend

- unit tests для util/store/composable;
- component tests для сложных UI-узлов;
- e2e tests для критичных сценариев.

### Bot

- unit tests для handlers;
- integration tests для webhook/internal endpoints.

## 11.2. Приоритет критичных сценариев

Тестами должны быть закрыты сначала:

- login;
- refresh/logout;
- создание заказа;
- смена статуса заказа;
- проверка прав доступа;
- восстановление сессии;
- cart/checkout flow;
- доставка по зоне.

## 11.3. Definition of Done для новой фичи

Фича считается готовой, если:

- код разбит по слоям;
- нет пустых `catch`;
- нет `console.*`;
- есть валидация входа;
- обновлены docs;
- добавлены тесты на критичный happy path;
- CI проходит;
- конфиги и `.env.example` обновлены.

## 12. Code review стандарт

Каждый PR должен быть:

- маленьким;
- с одной целью;
- с понятным названием;
- без смешения refactor и feature, если это можно разделить.

### PR template должен отвечать на 5 вопросов

1. Что изменено?
2. Зачем изменено?
3. Какие риски?
4. Как проверялось?
5. Что еще осталось сделать?

### Review checklist

Проверяющий обязан смотреть:

- понятность имен;
- смешение слоев;
- лишние зависимости;
- качество ошибок;
- соответствие структуре;
- тестируемость;
- обратную совместимость;
- соответствие docs.

## 13. Поддержание кодовой базы

## 13.1. Еженедельно

- смотреть большие файлы и новые god-files;
- проверять расхождения между README, docs, env и workflow;
- смотреть `npm audit` и уязвимые зависимости;
- смотреть дублирование кода.

## 13.2. Раз в 2 недели

- выделять 10–20% времени на техдолг;
- выносить повторяющийся код;
- упрощать длинные guards, stores, services;
- актуализировать документацию.

## 13.3. Раз в месяц

- проверять bundle size фронтов;
- смотреть покрытие тестами;
- пересматривать самые длинные файлы в репозитории;
- удалять мертвый код и неиспользуемые зависимости.

## 14. Запрещенные практики

Запрещено:

- пустые `catch`;
- `TODO` без задачи или срока;
- SQL внутри router/controller;
- бизнес-логика внутри Vue router guard;
- большой router-файл как единственная точка навигации;
- прямые сетевые запросы из больших UI-компонентов без отдельного API слоя;
- расхождения между документацией и реальным стеком;
- дублирование env-названий между приложением и workflow;
- смешение логирования через `console.*` и logger.

## 15. Рекомендуемый минимальный стек качества

Для текущей кодовой базы:

- ESLint
- Prettier
- EditorConfig
- Husky
- lint-staged
- Vitest для `admin-panel` и `telegram-miniapp`
- Supertest + Node test runner или Vitest для `backend`
- Playwright для критичных e2e
- schema validation: `zod` или `joi`

## 16. Целевое состояние через 1–2 итерации

Проект считается приведенным в порядок, если:

- нет файлов-контейнеров на 500+ строк без уважительной причины;
- router разбит на модульные route files;
- backend auth разделен по слоям;
- frontend использует lazy loading везде, где это нужно;
- README, docs, env и workflows совпадают;
- все приложения используют единый lint/format/test стандарт;
- есть покрытие тестами на критичные пользовательские потоки;
- новый разработчик может поднять проект по одной инструкции без ручной магии.
