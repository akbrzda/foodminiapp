## 3. Telegram Mini App (Клиентская часть)

### 3.1 Общие положения

Telegram Mini App является основным клиентским интерфейсом системы. Пользователь взаимодействует с приложением внутри Telegram без необходимости установки отдельного мобильного приложения. Mini App предназначен для оформления заказов, просмотра меню, участия в программе лояльности и просмотра истории заказов.
Приложение работает в online режиме с локальным кешированием критичных данных (меню, адреса, настройки).

---

### 3.2 Авторизация пользователя

Авторизация пользователя осуществляется автоматически через Telegram с использованием механизма передачи контакта.
**Логика работы:**

- Пользователю предлагается кнопка «Поделиться номером телефона»
- Telegram передаёт номер телефона в Mini App
- Номер телефона становится уникальным идентификатором пользователя в системе
  **Ограничения:**
- Без передачи номера телефона пользователь может просматривать меню и информацию
- Оформление заказа и использование бонусов недоступны
  **Хранимые данные пользователя:**
- номер телефона
- имя
- фамилия
- дата рождения
- email

---

### 3.3 Определение города и филиала

При первом запуске Mini App выполняется попытка определения города пользователя.
**Алгоритм:**

1. Запрос геолокации у пользователя
2. При успешном получении — определение города
3. При отказе — отображение списка доступных городов
   После выбора города пользователь вручную выбирает филиал (ресторан).
   Источник данных по городам и филиалам — локальная база данных. Данные кешируются в Redis (TTL: 1 час).

---

### 3.4 Меню

Меню отображается в разрезе выбранного города.
**Структура меню:**

- категории
- позиции
- модификаторы
  Каждая позиция может содержать:
- название
- описание
- цену
- изображение
- набор модификаторов
  **Источник данных:**
- Меню создается и редактируется через админ-панель
- Меню привязано к городу
- Позиции можно включать/отключать (наличие)
- Данные кешируются в Redis (TTL: 5 минут)
  **Варианты позиций:**
  Одна позиция может иметь несколько вариантов с разными ценами.
  Примеры:
- Пицца: Маленькая (25см) — 450₽, Средняя (30см) — 650₽, Большая (35см) — 850₽
- Картофель фри: Средняя порция — 150₽, Большая порция — 220₽
  **Модификаторы:**
  Типы модификаторов:
- **Одиночный выбор** (radio) — пользователь выбирает один вариант
- **Множественный выбор** (checkbox) — пользователь может выбрать несколько
  Обязательность:
- **Обязательный** — пользователь должен выбрать перед добавлением в корзину
- **Опциональный** — можно пропустить
  Примеры:
- Уровень прожарки стейка (обязательный, одиночный): Rare, Medium, Well done
- Дополнительные ингредиенты (опциональный, множественный): Сыр (+50₽), Бекон (+60₽)

---

### 3.5 Корзина и оформление заказа

Пользователь может добавить позиции меню в корзину и оформить заказ.
**Поддерживаемые типы заказа:**

- доставка
- самовывоз
  **Общие параметры заказа:**
- список позиций и модификаторов
- сумма заказа
- использование бонусов
- изменение количество позиций, удаление позиций
- комментарий к заказу
- желаемое время выполнения (только текущий день)
  По умолчанию используется значение «как можно скорее».

---

### 3.6 Доставка

При выборе доставки пользователю предлагается примерный адрес исходя из геолокации, после чего он указывает адрес.
**Форма адреса:**

- улица
- дом
- подъезд
- квартира
- домофон
- комментарий
  Адреса сохраняются в профиле пользователя. Поддерживается несколько адресов, один из которых может быть выбран по умолчанию.
  Проверка попадания адреса в зону доставки выполняется через PostGIS функцию `ST_Contains` с использованием полигонов, сохраненных в БД.
**Геокодирование адреса:**

1. Пользователь вводит адрес вручную
2. При вводе улицы показываются подсказки из локального справочника/поиска адресов системы
3. Пользователь видит только улицу и номер дома (например: "проспект Мира, 1")
4. Итоговый адрес (улица + дом) геокодируется через настроенный геокодинг-провайдер (преобразуется в координаты)
5. Проверяется попадание координат в полигон доставки выбранного филиала (PostGIS)
   **Результаты проверки:**

- **Адрес входит в зону:** пользователь заполняет детали (подъезд, квартира, домофон)
- **Адрес не входит в зону:** отображается ошибка с предложением изменить адрес
  **Ограничения Nominatim API:**
- Rate limit: 1 запрос в секунду
- Необходим User-Agent в запросах
- Результаты кешируются в Redis (TTL: 24 часа)

---

### 3.7 Самовывоз

При выборе самовывоза пользователь:

- использует ранее выбранный город
- выбирает филиал
- указывает время получения заказа

**Поддерживаемые способы:**

- банковская карта
  Онлайн-оплата в рамках текущего ТЗ не реализуется.

### 3.9 Программа лояльности

Система лояльности реализуется по требованиям `docs/bonus.md` и поддерживает неограниченное число уровней. Полное описание требований находится в `docs/bonus.md`.

**Уровни лояльности:**
- Уровни задаются в админ‑панели: название, порог, процент начисления, максимальный процент списания, активность.
- Порог всегда уникален; стартовый уровень имеет порог 0₽.
- Порядок уровней определяется автоматически по порогу.

**Глобальные настройки:**
- Включение/выключение системы лояльности.
- Период расчета суммы для уровня (по умолчанию 60 дней).
- Глобальный лимит списания бонусов от суммы заказа (по умолчанию 20%).
- Срок действия бонусов (по умолчанию 60 дней).
- Учитывать доставку в начислении (да/нет).
- Начислять от суммы после списания бонусов (да/нет).

**Начисление и списание:**
- Списание происходит при создании заказа, бонусы фиксируются как `spend` со статусом `pending`.
- Начисление происходит при первом статусе `delivered/completed`, создается `earn` со статусом `completed`.
- Сумма начисления фиксируется в заказе (`bonus_earn_amount`) и используется при повторной доставке.
- Округление всегда вниз.
- Списание работает по FIFO (сначала более старые бонусы).

**Исключения:**
- Исключения задаются по категориям и товарам и влияют только на списание.
- Если весь заказ состоит из исключенных позиций, списание недоступно.

**Автоматические начисления и деградация:**
- Бонус за регистрацию и бонус ко дню рождения (настраиваемые параметры).
- Деградация уровня при неактивности (пошаговая, период настраивается).

**Отображение в Mini App:**
- Баланс бонусов, текущий уровень и прогресс до следующего.
- Скоро сгорят (истекающие бонусы).
- История транзакций с типами и сроками действия.

**Хранение и кеш:**
- Баланс хранится в `users.bonus_balance`.
- История транзакций — `loyalty_transactions`.
- Баланс кешируется в Redis (TTL: 1 минута).

## 4. Админ-панель

### 4.1 Назначение админ-панели

Админ-панель является рабочим инструментом персонала и управляющего состава. В условиях недоступности основной системы админ-панель становится **единственным источником информации о заказах** и используется для фактической работы кухни и менеджеров.
Админ-панель реализуется как веб-приложение и работает в режиме реального времени.

---

### 4.2 Роли и доступы

**Администратор**

- просматривает аналитику и заказы по всем городам (меняя в фильтре)
- настраивает систему, меняет настройки
- управляет пользователями (админами и менеджерами)
- просматривает полигоны доставки
- редактирует меню (MVP: локально)
- просматривает очереди фоновых задач (уведомления)
- управляет настройками системы
- ведёт аудит действий админов
  **Менеджер**
- доступ только к своим городам
- просматривает аналитику и заказы по своим городам
- редактирует меню (MVP: локально)
- просматривает полигоны доставки
  **CEO**
- просмотр всех городов без технических разделов
- доступ к аналитике и заказам по всем городам
- просмотр полигона доставки
- редактирует меню (MVP: локально)

---

### 4.3 Работа с заказами

Админ-панель отображает список заказов в режиме реального времени.
**Функциональность:**

- автоматическое обновление списка заказов
- визуальное выделение новых заказов
  **Фильтры:**
- По статусу заказа
- По городу/филиалу
- По дате (сегодня, вчера, неделя, месяц, произвольный период)
- По типу заказа (доставка/самовывоз)
- По способу оплаты
  **Изменение статусов:**
  Доступные статусы:

1. Новый (желтый бейдж)
2. Принят (синий)
3. Готовится (оранжевый)
4. В пути / Готов к выдаче (фиолетовый)
5. Доставлен / Выдан (зеленый)
6. Отменен (красный)

### 4.6 Полигоны доставки

Раздел предназначен для управления зонами доставки с использованием интерактивной карты.
**Функциональность:**

- Карта с отображением полигонов (Leaflet.js + OpenStreetMap)
- Инструмент рисования полигонов (Leaflet.draw)
- Импорт/экспорт полигонов в формате GeoJSON
- Создание, редактирование, удаление полигонов
  **Отображается:**
- Список филиалов с выбором текущего
- Карта с полигонами текущего филиала
- Панель управления:
  - Кнопка "Добавить полигон"
  - Список существующих полигонов с кнопками редактирования/удаления
    **Хранение:**
- Полигоны хранятся в таблице `delivery_polygons`
- Тип поля: `POLYGON` (PostGIS)
- Координаты в формате WGS84 (SRID 4326)
  **Проверка адреса:**
  Используется PostGIS функция:

````sql
SELECT ST_Contains(polygon, ST_SetSRID(ST_MakePoint(lng, lat), 4326))
FROM delivery_polygons
WHERE branch_id = ?
   ``

## 5. Backend и очереди задач
### 5.1 Общая логика
Backend реализует стандартный REST API с использованием очередей для фоновых задач.
**Основные компоненты:**
- Express.js сервер (API endpoints)
- WebSocket сервер (real-time обновления)
- Redis (кеширование + очереди)
- MySQL с PostGIS (основное хранилище)
### 5.2 Очереди задач (Redis Queue)
Используется Redis для организации очередей фоновых задач.
**Типы задач:**
1. **Telegram уведомления**
   - Уведомление персонала о новом заказе
   - Критичные ошибки системы
2. **Обработка изображений**
   - Оптимизация загруженных изображений меню
   - Конвертация в WebP
   - Создание thumbnails
3. **Расчет аналитики** (опционально)
   - Агрегация данных для отчетов
   - Обновление статистики
**Структура очереди:**
```javascript
await redisClient.lpush('queue:telegram', JSON.stringify({
  type: 'new_order',
  orderId: 1234,
  data: {...}
}));
const task = await redisClient.brpop('queue:telegram', 0);
await processTelegramNotification(task);
````

**Обработка ошибок:**

- При ошибке задача перемещается в очередь повторных попыток
- Максимум 3 попытки с экспоненциальной задержкой (1 мин, 5 мин, 15 мин)
- После 3 неудачных попыток задача перемещается в dead-letter queue
- Администратор видит failed tasks в админ-панели

### 5.3 Кеширование (Redis Cache)

**Что кешируется:**
| Данные | TTL | Ключ Redis |
| ------------------------- | -------- | ------------------------ |
| Меню города | 5 минут | `menu:city_{id}` |
| Баланс бонусов | 1 минута | `bonuses:user_{id}` |
| Настройки системы | 10 минут | `settings` |
| Настройки лояльности | 10 минут | `loyalty_settings` |
| Уровни лояльности | 5 минут | `loyalty_levels` |
| Полигоны филиала | 1 час | `polygons:branch_{id}` |
| Результаты геокодирования | 24 часа | `geocode:{address_hash}` |
**Инвалидация кеша:**

- При изменении меню → удаление ключа `menu:city_{id}`

- При изменении настроек → удаление ключа `settings`
- При изменении настроек лояльности → удаление ключа `loyalty_settings`
- При изменении уровней лояльности → удаление ключа `loyalty_levels`
- При изменении полигонов → удаление ключа `polygons:branch_{id}`

## 8. Ограничения и зависимости

### 8.1 Технические ограничения

**Nominatim API (OpenStreetMap):**

- Rate limit: 1 запрос в секунду
- Требуется User-Agent в заголовках
- Может быть недоступен (кешировать результаты)
- Качество данных зависит от региона
  **Address Directory / Geocoding API:**
- Используется только для подсказок адреса и геокодирования
- Не зависит от интеграции с iiko
  **iiko API:**
- Используется для меню, стоп-листа и заказов
- Не используется как справочник адресов, городов и филиалов
  **PostGIS:**
- Требует расширение в MySQL 8.0+
- Более сложная настройка, чем обычный MySQL
- Индексы SPATIAL INDEX для производительности
  **Telegram Bot API:**
- Лимиты на отправку сообщений: 30 сообщений/секунду на бота
- Может быть заблокирован пользователем
- Зависимость от доступности Telegram
  **Redis:**
- Хранение данных в RAM (требуется достаточно памяти)
- При перезапуске данные могут быть потеряны (настроить persistence)

### 8.2 Бизнес-ограничения

- Количество полигонов доставки на филиал не ограничено
- Минимальная сумма заказа: задается в настройках
- Максимальное списание бонусов: настраиваемый лимит (по умолчанию 20% от суммы заказа)
- Время доставки: только "как можно скорее" в MVP

### 8.3 Зависимости от внешних сервисов

**Критичные:**

- Telegram (для Mini App)
- OpenStreetMap/Nominatim (для геокодирования адресов)
- iiko API (для меню/стоп-листа/заказов)

### 8.4 Риски

| Риск                    | Вероятность | Влияние | Митигация                                                    |
| ----------------------- | ----------- | ------- | ------------------------------------------------------------ |
| Недоступность Nominatim | Средняя     | Высокое | Кеширование результатов, fallback на сохраненный адрес/карту |
| Недоступность сервиса геокодинга/подсказок | Средняя | Среднее | Переключение на резервный провайдер и использование локального кеша |
| Рост нагрузки           | Высокая     | Среднее | pm2 cluster mode, Redis кеширование, индексы БД              |
| Падение сервера         | Низкая      | Высокое | pm2 автоперезапуск, мониторинг, резервные копии              |
| Изменение Telegram API  | Низкая      | Высокое | Использование стабильных методов API, подписка на обновления |
