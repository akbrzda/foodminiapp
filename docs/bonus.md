## Общее описание

### Цель проекта

Разработка упрощенной бонусной системы лояльности для онлайн-заказа еды с фиксированными параметрами для быстрого запуска MVP.

### Основные возможности

- 3 фиксированных уровня лояльности
- Автоматическое начисление и списание бонусов
- Автоматическое управление уровнями пользователей
- Автоначисления по триггерам (регистрация, день рождения)
- Истечение бонусов
- Детальная история транзакций
- Минималистичная административная панель
- Доступ через настройки системы во вкладке Лояльность (убрать отдельную страницу)

## Параметры системы

### Фиксированные уровни лояльности

Система включает 3 уровня, захардкоженных в конфигурации:

| Уровень | Порог достижения | Процент начисления | Макс. процент списания |
|---------|------------------|-------------------|----------------------|
| Бронза  | 0₽              | 3%                | 25%                  |
| Серебро | 10,000₽         | 5%                | 25%                  |
| Золото  | 20,000₽         | 7%                | 25%                  |

**Правила:**
- Первый уровень (Бронза) доступен всем с момента регистрации
- Переход на следующий уровень происходит автоматически при достижении порога
- Порог считается по сумме завершенных заказов за всё время
- Максимальный процент списания одинаков для всех уровней

### Автоматические начисления

**Бонус за регистрацию:**
- Сумма: 1,000 бонусов
- Начисляется: один раз при первой регистрации в программе лояльности
- Срок действия: 30 дней с момента начисления

**Бонус за день рождения:**
- Сумма: 500 бонусов
- Начисляется: за 3 дня до дня рождения
- Срок действия: 7 дней со дня рождения
- Периодичность: один раз в год

Пример расчета срока ДР бонуса:
- День рождения: 25 января
- Начисление: 22 января (за 3 дня)
- Истекает: 1 февраля (25 января + 7 дней)

### Глобальные параметры

**Фиксированные настройки (захардкожены):**
- Срок действия начисленных бонусов: 60 дней
- Период расчета суммы для уровня: за всё время
- Округление: всегда вниз (floor)
- Деградация уровней: отключена

**Единственная настройка с UI:**
- Включение/выключение модуля лояльности

### Формулы расчета

**Начисление бонусов:**
- База для начисления = Итоговая сумма заказа - Стоимость доставки - Использованные бонусы
- Сумма начисления = База × Процент начисления уровня
- Округление вниз до целого рубля

Пример:
```
Заказ: 2,000₽
Доставка: 200₽
Использовано бонусов: 300₽
Уровень: Серебро (5%)

Расчет: (2,000 - 200 - 300) × 0.05 = 1,500 × 0.05 = 75₽
Начислится: 75₽
```

**Списание бонусов:**
- База для списания = Итоговая сумма заказа - Стоимость доставки
- Максимум списания = База × 25%
- Округление вниз до целого рубля

Пример:
```
Заказ: 2,000₽
Доставка: 200₽

Расчет: (2,000 - 200) × 0.25 = 1,800 × 0.25 = 450₽
Максимум для списания: 450₽
```

## Архитектура базы данных

### Таблица: loyalty_levels

Хранит информацию об уровнях лояльности (заполняется один раз при миграции).

**Поля:**
- `id` (INTEGER, PK, AUTO_INCREMENT)
- `name` (VARCHAR(50)) - название уровня
- `threshold_amount` (DECIMAL(10,2)) - порог достижения в рублях
- `earn_percentage` (INTEGER) - процент начисления (3, 5, 7)
- `max_spend_percentage` (INTEGER) - максимальный процент списания (25)
- `is_enabled` (BOOLEAN, default: true)
- `created_at` (TIMESTAMP)
- `updated_at` (TIMESTAMP)

**Индексы:**
- PRIMARY KEY на `id`
- UNIQUE на `threshold_amount`

**Данные (фиксированные):**
```
id=1: Бронза,  порог=0,     начисление=3%, списание=25%
id=2: Серебро, порог=10000, начисление=5%, списание=25%
id=3: Золото,  порог=20000, начисление=7%, списание=25%
```

---

### Таблица: users (изменения)

Добавляются поля для интеграции с системой лояльности.

**Новые поля:**
- `current_loyalty_level_id` (INTEGER, FK → loyalty_levels.id) - текущий уровень пользователя
- `loyalty_balance` (DECIMAL(10,2), default: 0.00) - текущий баланс бонусов
- `loyalty_joined_at` (TIMESTAMP, nullable) - дата вступления в программу

**Индексы:**
- INDEX на `current_loyalty_level_id`
- INDEX на `loyalty_joined_at`

---

### Таблица: orders (изменения)

Добавляются поля для отслеживания бонусных операций.

**Новые поля:**
- `bonus_spent` (DECIMAL(10,2), default: 0.00) - сумма использованных бонусов
- `bonus_earn_amount` (DECIMAL(10,2), nullable) - зафиксированная сумма начисления
- `bonus_earn_locked` (BOOLEAN, default: false) - флаг защиты от дублей

**Индексы:**
- INDEX на `bonus_earn_locked`
- INDEX на (`user_id`, `status`, `created_at`) для расчета уровня

---

### Таблица: loyalty_transactions

Хранит историю всех операций с бонусами.

**Поля:**
- `id` (INTEGER, PK, AUTO_INCREMENT)
- `user_id` (INTEGER, FK → users.id)
- `type` (ENUM) - тип транзакции
  - `earn` - начисление
  - `spend` - списание
  - `expire` - истечение
  - `registration` - бонус за регистрацию
  - `birthday` - бонус за день рождения
  - `adjustment` - ручная корректировка
- `status` (ENUM)
  - `pending` - ожидает (для списаний при статусе new)
  - `completed` - выполнена
  - `cancelled` - отменена
- `amount` (DECIMAL(10,2)) - сумма транзакции
- `remaining_amount` (DECIMAL(10,2)) - остаток (для типа earn)
- `order_id` (INTEGER, FK → orders.id, nullable)
- `related_transaction_id` (INTEGER, FK → loyalty_transactions.id, nullable)
- `description` (TEXT, nullable) - описание операции
- `expires_at` (TIMESTAMP, nullable) - дата истечения (для earn)
- `admin_id` (INTEGER, FK → admins.id, nullable) - кто выполнил (для adjustment)
- `created_at` (TIMESTAMP)
- `updated_at` (TIMESTAMP)

**Индексы:**
- PRIMARY KEY на `id`
- INDEX на `user_id`
- INDEX на `order_id`
- INDEX на `type`
- INDEX на `status`
- INDEX на `expires_at` (для поиска истекающих)
- COMPOSITE на (`user_id`, `type`, `created_at`) для истории

**Логика FIFO при списании:**
- При списании бонусов списываются сначала самые старые начисления (по created_at)
- У каждого начисления (type=earn) есть remaining_amount
- При частичном использовании уменьшается remaining_amount

---

**Поля:**

**Индексы:**
- PRIMARY KEY на `id`
- INDEX на `user_id`
- INDEX на (`user_id`, `created_at`) для истории

---

### Таблица: loyalty_logs

Технические логи для отладки и аудита.

**Поля:**
- `id` (INTEGER, PK, AUTO_INCREMENT)
- `user_id` (INTEGER, FK → users.id, nullable)
- `order_id` (INTEGER, FK → orders.id, nullable)
- `event_type` (VARCHAR(50)) - тип события
  - `balance_calculated` - расчет баланса
  - `level_changed` - смена уровня

**Индексы:**
- PRIMARY KEY на `id`

- INDEX на `order_id`
- INDEX на `event_type`
- INDEX на `created_at`

## Бизнес-логика

### Вступление в программу лояльности

**Триггер:** Регистрация нового пользователя

**Действия:**
1. Установить `current_loyalty_level_id = 1` (Бронза)
2. Установить `loyalty_joined_at = NOW()`
3. Начислить 1,000 бонусов (тип: registration)
4. Установить срок истечения: NOW() + 30 дней
5. Создать запись в `user_loyalty_levels`

---

### Начисление бонусов

**Момент начисления:** При первом достижении заказом статуса `delivered` или `completed`

**Условия:**
- Заказ должен быть в статусе delivered/completed
- Поле `bonus_earn_locked = false` (защита от дублей)
- Пользователь участвует в программе

**Алгоритм:**

1. **Проверка блокировки:**
   - Если `bonus_earn_locked = true` → пропустить начисление
   - Установить `bonus_earn_locked = true` атомарно

2. **Расчет суммы:**
   - База = `order_total - delivery_cost - bonus_spent`
   - Процент = процент начисления текущего уровня пользователя
   - Сумма = `floor(База × Процент)`

3. **Фиксация суммы:**
   - Сохранить рассчитанную сумму в `orders.bonus_earn_amount`
   - Даже если уровень изменится позже - использовать эту зафиксированную сумму

4. **Создание транзакции:**
   - Тип: `earn`
   - Статус: `completed`
   - Сумма: из `bonus_earn_amount`
   - Срок истечения: NOW() + 60 дней
   - remaining_amount = amount

5. **Обновление баланса:**
   - Увеличить `users.loyalty_balance` на сумму начисления

**Защита от дублирования:**
- Флаг `bonus_earn_locked` предотвращает повторное начисление
- При повторном переходе в delivered используется уже зафиксированная сумма
- Проверка affected_rows при установке блокировки

---

### Списание бонусов

**Момент списания:** При создании заказа (статус `new`)

**Условия:**
- Пользователь указывает количество бонусов для использования
- Баланс пользователя достаточен
- Не превышен максимальный процент списания (25%)

**Алгоритм:**

1. **Валидация:**
   - Проверить баланс: `loyalty_balance >= запрошенная_сумма`
   - Рассчитать максимум: `floor((order_total - delivery_cost) × 0.25)`
   - Проверить: `запрошенная_сумма <= максимум`

2. **Выбор бонусов по FIFO:**
   - Выбрать активные начисления (type=earn, status=completed, remaining_amount > 0)
   - Отсортировать по created_at (от старых к новым)
   - Списывать последовательно с самых старых

3. **Создание транзакции:**
   - Тип: `spend`
   - Статус: `pending` (до подтверждения заказа)
   - Сумма: запрошенная сумма
   - Привязка к order_id

4. **Уменьшение remaining_amount:**
   - У каждого использованного earn уменьшить remaining_amount
   - Если бонус использован полностью - remaining_amount = 0

5. **Обновление баланса:**
   - Уменьшить `users.loyalty_balance` на сумму списания

6. **Сохранение в заказе:**
   - Установить `orders.bonus_spent = сумма`

**Подтверждение при delivered:**
- Изменить статус транзакции spend на `completed`
- Никаких дополнительных действий не требуется

---

### Отмена списания (cancelled заказ)

**Момент:** При переходе заказа в статус `cancelled`

**Условия:**
- В заказе были использованы бонусы (`bonus_spent > 0`)
- Есть транзакция типа `spend` со статусом `pending`

**Алгоритм:**

1. **Отмена транзакции:**
   - Найти транзакцию spend для этого заказа
   - Изменить статус на `cancelled`

2. **Возврат бонусов:**
   - Восстановить remaining_amount у earn транзакций (по связи)
   - Увеличить `users.loyalty_balance` на сумму возврата

3. **Обнуление в заказе:**
   - Установить `orders.bonus_spent = 0`

**Важно:** Возврат происходит к тем же earn транзакциям, откуда были списаны, сохраняя FIFO очередность.

---

### Управление уровнями пользователей

**Триггер:** После создания транзакции earn (начисление за заказ)

**Частота проверки:** После каждого завершенного заказа

**Алгоритм:**

1. **Расчет суммы заказов:**
   - Выбрать все завершенные заказы пользователя (за всё время)
   - Статусы: `delivered` или `completed`
   - Сумма = `SUM(order_total - delivery_cost - bonus_spent)`

2. **Определение уровня:**
   - Сравнить сумму с порогами уровней (0₽, 10,000₽, 20,000₽)
   - Выбрать максимальный уровень, порог которого <= сумма

3. **Изменение уровня (если нужно):**
   - Если новый уровень отличается от текущего:
     - Обновить `users.current_loyalty_level_id`
     - Создать запись в `user_loyalty_levels`
     - Указать previous_level_id, reason, threshold_sum

**Правила:**
- Переход возможен только вверх (деградация отключена в MVP)
- Уровень пересчитывается только при начислении за заказ
- При ручной корректировке баланса уровень НЕ меняется

---

### Истечение бонусов

**Триггер:** Cron задача, выполняется ежедневно в 04:00

**Алгоритм:**

1. **Поиск истекших:**
   - Выбрать все earn транзакции
   - Условия:
     - `status = completed`
     - `expires_at < NOW()`
     - `remaining_amount > 0`

2. **Создание транзакций expire:**
   - Для каждого истекшего начисления:
     - Создать новую транзакцию
     - Тип: `expire`
     - Статус: `completed`
     - Сумма: remaining_amount
     - related_transaction_id: ссылка на earn

3. **Обновление балансов:**
   - Уменьшить `users.loyalty_balance` на сумму истекших
   - Обнулить remaining_amount у earn транзакций

4. **Логирование:**
   - Записать в loyalty_logs событие `bonus_expired`

---

### Бонус за день рождения

**Триггер:** Cron задача, выполняется ежедневно в 02:00

**Алгоритм:**

1. **Определение целевой даты:**
   - Рассчитать дату = NOW() + 3 дня
   - Извлечь день и месяц

2. **Поиск пользователей:**
   - Выбрать пользователей с birth_date
   - День и месяц birth_date совпадают с целевой датой
   - У пользователя нет транзакции birthday в текущем году

3. **Начисление бонуса:**
   - Создать транзакцию:
     - Тип: `birthday`
     - Статус: `completed`
     - Сумма: 500₽
     - Срок истечения: день рождения + 7 дней
     - remaining_amount = 500

4. **Обновление баланса:**
   - Увеличить `users.loyalty_balance` на 500₽

**Пример:**
- Сегодня: 22 января
- Целевая дата: 25 января (22 + 3 дня)
- Найти пользователей с ДР 25 января
- Начислить 500₽, срок: 1 февраля (25 января + 7 дней)

---

### Корректировка баланса администратором

**Триггер:** Ручное действие администратора в админ панели

**Типы корректировок:**
- Начисление (adjustment с положительной суммой)
- Списание (adjustment с отрицательной суммой)

**Алгоритм:**

1. **Валидация:**
   - Указан пользователь
   - Указана сумма (может быть положительной или отрицательной)
   - Указана причина корректировки
   - При списании: баланс >= сумма списания

2. **Создание транзакции:**
   - Тип: `adjustment`
   - Статус: `completed`
   - Сумма: указанная сумма
   - Описание: причина
   - admin_id: ID администратора

3. **Для начисления (+):**
   - Установить expires_at: NOW() + 60 дней
   - remaining_amount = amount
   - Увеличить `users.loyalty_balance`

4. **Для списания (-):**
   - Списать по FIFO (как при обычном списании)
   - Уменьшить remaining_amount у earn транзакций
   - Уменьшить `users.loyalty_balance`

5. **Логирование:**
   - Записать в loyalty_logs событие `balance_adjusted`
   - Сохранить old_value и new_value баланса

**Важно:** Корректировка НЕ влияет на расчет уровня пользователя.

## API спецификация

### Клиентские эндпоинты

#### Получить баланс и информацию о программе

**GET** `/api/client/loyalty/balance`

**Авторизация:** Требуется (JWT токен пользователя)

**Ответ:** Объект с данными о балансе, текущем уровне, прогрессе до следующего уровня, истекающих бонусах

---

#### Рассчитать максимальное списание для заказа

**GET** `/api/client/loyalty/calculate-max-spend`

**Параметры запроса:**
- `orderTotal` - итоговая сумма заказа
- `deliveryCost` - стоимость доставки

**Авторизация:** Требуется

**Логика:**
- База = orderTotal - deliveryCost
- Максимум = floor(База × 0.25)
- Проверка баланса пользователя
- Возврат минимума из (максимум, баланс)

**Ответ:** Объект с максимальной суммой для списания

---

#### Получить историю транзакций

**GET** `/api/client/loyalty/history`

**Параметры запроса:**
- `page` - номер страницы (default: 1)
- `limit` - количество записей (default: 50, max: 50)

**Авторизация:** Требуется

**Ответ:** Список транзакций с информацией о типе, сумме, дате, номере заказа, сроке истечения. Флаг наличия следующей страницы.

---

#### Получить список уровней лояльности

**GET** `/api/client/loyalty/levels`

**Авторизация:** Требуется

**Ответ:** Текущий уровень пользователя, список всех уровней с порогами и процентами, прогресс до следующего уровня.

---

### Административные эндпоинты

#### Получить статус модуля лояльности

## Технические требования

### Производительность

**Кеширование:**
- Баланс пользователя: Redis, TTL 60 секунд
  - Ключ: `loyalty:balance:{userId}`
  - Инвалидация: при любой транзакции пользователя
- Настройки модуля: Redis, TTL 10 минут
  - Ключ: `loyalty:settings`
  - Инвалидация: при изменении настроек
- Фиксированные уровни: Redis, без TTL (постоянно)
  - Ключ: `loyalty:levels`
  - Загружается один раз при старте

**Индексы БД:**
- Все внешние ключи проиндексированы
- Composite индексы для частых запросов:
  - `(user_id, status, created_at)` на orders
  - `(user_id, type, created_at)` на loyalty_transactions
  - `(type, status, expires_at)` на loyalty_transactions
- Индексы на поля фильтрации и сортировки

**Оптимизация запросов:**
- Использование JOIN вместо множественных запросов
- Агрегация на уровне БД (SUM, COUNT)
- Пагинация для больших списков (history)
- SELECT только необходимых полей

---

### Безопасность

**Аутентификация и авторизация:**
- JWT токены для клиентов
- Проверка ролей для админских эндпоинтов
- Rate limiting на API:
  - Клиентские эндпоинты: 100 запросов/минуту
  - Админские эндпоинты: 200 запросов/минуту

**Валидация:**
- Все входные данные валидируются на backend
- Проверка диапазонов для числовых полей
- Защита от SQL injection (параметризованные запросы)
- Защита от XSS (экранирование вывода)
- Проверка прав доступа к данным пользователя

**Транзакции БД:**
- Все операции с бонусами выполняются в транзакциях
- Уровень изоляции: READ COMMITTED для большинства операций
- SERIALIZABLE для критичных операций (корректировка баланса)
- Использование SELECT ... FOR UPDATE для предотвращения race conditions

**Логирование:**
- Все изменения балансов логируются в loyalty_logs
- Логирование подозрительной активности (отрицательные балансы, попытки превысить лимиты)
- Запись администратора выполнившего действие (для adjustment)
- Сохранение старых и новых значений при изменениях

---

### Надежность

**Обработка ошибок:**
- Try-catch для всех операций с БД
- Откат транзакций при ошибках
- Информативные сообщения об ошибках для пользователя
- Детальное логирование ошибок для разработчиков

**Защита от race conditions:**
- Флаг блокировки `bonus_earn_locked` в заказах
- Atomic операции обновления (UPDATE ... WHERE)
- Проверка affected_rows после UPDATE
- Retry логика при конфликтах

**Мониторинг:**
- Логирование дублей транзакций в loyalty_logs
- Алерты при обнаружении отрицательных балансов
- Отслеживание расхождений баланса (сумма транзакций vs loyalty_balance)
- Метрики производительности API эндпоинтов

**Cron задачи:**
- Истечение бонусов: ежедневно в 04:00
  - Timeout: 30 минут
  - Batch обработка по 1000 записей
  - Логирование результатов
- Обработка ДР бонусов: ежедневно в 02:00
  - Timeout: 15 минут
  - Проверка дубликатов
  - Логирование начислений

### Масштабируемость

- Redis для часто запрашиваемых данных

- Предзагрузка фиксированных уровней

## Возможные риски и решения

### Риск 1: Дублирование начисления

**Проблема:** При нестабильной сети или ошибках возможно повторное начисление бонусов за один заказ.

**Решение:**
- Флаг `bonus_earn_locked` блокирует повторные начисления
- Атомарная операция установки флага
- Проверка affected_rows
- Логирование попыток дублирования

---

### Риск 2: Race condition при списании

**Проблема:** Два параллельных заказа могут списать больше бонусов, чем доступно на балансе.

**Решение:**
- Транзакции БД с уровнем изоляции SERIALIZABLE
- SELECT ... FOR UPDATE при чтении баланса
- Проверка баланса перед каждым списанием
- Откат транзакции при недостаточном балансе

---

### Риск 3: Производительность cron задач

**Проблема:** При большом количестве пользователей cron задачи могут выполняться долго.

**Решение:**
- Batch обработка по 1000 записей
- Timeout на выполнение задачи
- Логирование прогресса
- Оптимизированные запросы с индексами

---

### Риск 4: Несоответствие баланса и транзакций

**Проблема:** Из-за ошибок баланс может не совпадать с суммой транзакций.

**Решение:**
- Периодическая проверка целостности (можно запускать вручную)
