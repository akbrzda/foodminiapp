# FoodMiniApp — roadmap приведения кодовой базы в порядок и улучшения читаемости

Версия: 1.0  
Горизонт: 6 недель  
Формат: практический план стабилизации, рефакторинга и выравнивания инженерной дисциплины

---

## 1. Цели roadmap

К концу roadmap кодовая база должна получить:

- единый инженерный стандарт;
- настоящие quality gates;
- согласованную документацию и конфиги;
- уменьшение числа перегруженных файлов;
- более читаемую структуру frontend и backend;
- предсказуемые правила для ревью и поддержки.

---

## 2. Что считаем успехом

### Измеримые критерии

#### Репозиторий и процессы

- `README`, `.env.example`, workflow env и реальные зависимости согласованы;
- у каждого сервиса есть реальные `lint`, `typecheck`, `test`;
- pre-commit запускает базовые проверки.

#### Размер и читаемость

- ни один критичный файл не превышает 400 строк;
- router admin-panel разбит на route modules и guards;
- backend auth разбит на слои;
- в miniapp router больше не является центром всей UI-оркестрации.

#### Качество

- нет новых silent catch;
- `console.*` заменён централизованным подходом там, где это нужно;
- добавлены тесты на критические сценарии.

---

## 3. Приоритеты

### P0 — срочно

Это нужно сделать первым, иначе любой дальнейший рефакторинг будет строиться на шаткой основе.

- выровнять документацию и конфиги;
- ввести реальный lint/test/typecheck контур;
- зафиксировать стандарты.

### P1 — высокий приоритет

- разрезать `backend/src/modules/auth/routes.js`;
- разрезать `admin-panel/src/router/index.js`;
- перевести admin routes на lazy loading;
- вынести frontend guards в отдельные файлы.

### P2 — средний приоритет

- привести miniapp router и Telegram integration к более чистой структуре;
- выровнять stores;
- стандартизировать logging и error handling в `bot`.

### P3 — плановое улучшение

- добавить больше тестов;
- ввести архитектурные проверки и bundle-контроль;
- постепенно усиливать типизацию.

## 4. План по неделям

## Неделя 1. Зафиксировать правду проекта и поставить инженерный фундамент

### Цель

Убрать расхождения между кодом, документацией, env и CI.  
Поставить реальные правила качества.

### Задачи

#### 4.1.1. Документация и конфиги

- переписать `README.md` так, чтобы он описывал реальный стек;
- проверить все `.env.example`;
- выровнять workflow secrets и env-переменные с примерами;
- убрать ложные или спорные заявления о технологиях, если они фактически не используются.

#### 4.1.2. Tooling

Подключить:

- `eslint`
- `prettier`
- `.editorconfig`
- `husky`
- `lint-staged`

Для frontend:

- `vitest`
- `@vue/test-utils`

Для backend и bot:

- `supertest`
- единый test bootstrap

#### 4.1.3. Root-level стандартизация

Создать корневые файлы:

- `CONTRIBUTING.md`
- `.editorconfig`
- `.prettierrc`
- `eslint.config.js` или эквивалент;
- root `package.json` с общими командами;
- при возможности — workspace setup.

### Результат недели

- проект запускается с понятными командами;
- качество кода проверяется не формально, а реально;
- документация больше не спорит с кодом.

### Критерий готовности

- каждый сервис имеет работающий `lint`;
- `typecheck` больше не сводится к `node --check`;
- `.env.example` совпадают с тем, что реально нужно для запуска и деплоя.

## Неделя 2. Привести admin-panel к читаемой структуре

### Цель

Убрать главный frontend god-file и снизить связность роутера.

### Задачи

#### 4.2.1. Разделение router

Создать:

```text
admin-panel/src/router/
  index.js
  guards/
    auth.guard.js
    permission.guard.js
    navigation-context.guard.js
  routes/
    dashboard.routes.js
    orders.routes.js
    menu.routes.js
    users.routes.js
    loyalty.routes.js
    broadcasts.routes.js
    settings.routes.js
```

#### 4.2.2. Lazy loading

Все view-компоненты страниц перевести на:

```js
component: () => import("@/modules/.../views/SomePage.vue");
```

#### 4.2.3. Guards

Вынести из router:

- restore session;
- permission checks;
- navigation context logic.

#### 4.2.4. Store cleanup

- проверить `auth` store на избыточные обязанности;
- оставить только auth-related state/actions;
- побочные UI-эффекты вынести отдельно.

### Результат недели

- router читается как карта приложения, а не как смешанный runtime-файл;
- меньше initial bundle pressure;
- guard-логика тестируема отдельно.

### Критерий готовности

- `admin-panel/src/router/index.js` становится коротким файлом-композитором;
- маршруты разделены по доменам;
- нет тяжёлых статических импортов экранов без причины.

---

## Неделя 3. Разрезать backend auth и привести backend к строгому layering

### Цель

Разбить самый опасный технический узел на предсказуемые слои.

### Задачи

#### 4.3.1. Auth module refactor

Построить структуру:

```text
backend/src/modules/auth/
  auth.routes.js
  auth.controller.js
  auth.service.js
  auth.repository.js
  auth.schemas.js
  auth.tokens.js
  auth.cookies.js
  auth.mapper.js
```

#### 4.3.2. Вынести из route layer

- SQL-запросы;
- генерацию токенов;
- cookie handling helpers;
- permission mapping;
- validation;
- response normalization.

#### 4.3.3. Единый error model

Создать:

```text
backend/src/shared/errors/
  domain-error.js
  auth-errors.js
  validation-errors.js
```

#### 4.3.4. Logger cleanup

- заменить случайные `console.*` на единый logger там, где это касается backend;
- определить, что логируется на `info`, `warn`, `error`.

### Результат недели

- auth больше не монолитный файл;
- backend слои читаются и тестируются отдельно;
- сокращается риск поломать всё одной правкой.

### Критерий готовности

- route-файл auth не содержит SQL и крупной бизнес-логики;
- каждый слой можно объяснить одной фразой;
- базовые auth-сценарии покрыты тестами.

## Неделя 4. Привести Telegram Mini App к более чистой оркестрации

### Цель

Оставить сильный UX, но убрать хрупкость ручной логики.

### Задачи

#### 4.4.1. Router decomposition

Создать:

```text
telegram-miniapp/src/router/
  index.js
  guards/
    auth.guard.js
    location.guard.js
    telegram-back-button.guard.js
    scroll-restore.guard.js
```

#### 4.4.2. Telegram service adapter

Создать единый слой:

```text
telegram-miniapp/src/shared/services/telegram/
  telegram-client.js
  telegram-ui.js
  telegram-session.js
```

#### 4.4.3. Store cleanup

- привести store-стиль к единому соглашению;
- убрать silent catch;
- оставить только предметные обязанности.

#### 4.4.4. Local/session storage policy

- вынести ключи хранения в constants;
- добавить безопасное чтение и очистку;
- описать поведение при отсутствии Telegram WebApp context.

### Результат недели

- сильный UX сохраняется;
- router становится легче читать;
- Telegram API используется из одной точки.

### Критерий готовности

- router содержит только маршруты и подключение guard-модулей;
- UI-логика Telegram не размазана по приложению;
- ошибки не глотаются молча.

## Неделя 5. Упорядочить bot service и выровнять инфраструктурный стиль

### Цель

Подтянуть bot до уровня backend по инженерной дисциплине.

### Задачи

#### 4.5.1. Валидация

Добавить schema validation для:

- internal API;
- webhook payload boundaries;
- message-related input.

#### 4.5.2. Структура модулей

Разделить:

- webhook handling;
- internal endpoints;
- command bot logic;
- queue jobs;
- Telegram API helpers.

#### 4.5.3. Error and logging policy

- определить единый logger contract;
- убрать случайные `console.*`;
- нормализовать ответы API.

#### 4.5.4. Tests

Добавить smoke/integration tests на:

- `/health`;
- webhook secret validation;
- internal token protection;
- основные internal endpoints.

### Результат недели

- bot не отстаёт по дисциплине;
- снижается количество ручной неструктурированной логики;
- сервис становится легче сопровождать.

### Критерий готовности

- internal routes валидируются;
- ошибки видны и читаемы;
- тесты подтверждают базовую работоспособность.

## Неделя 6. Финальное выравнивание и контроль поддерживаемости

### Цель

Сделать так, чтобы улучшения закрепились и не откатились через месяц.

### Задачи

#### 4.6.1. Codebase cleanup pass

- убрать мёртвый код;
- удалить устаревшие helpers;
- привести импорт-порядок;
- выровнять naming;
- устранить повторяющиеся паттерны.

#### 4.6.2. Metrics and guardrails

Зафиксировать технические метрики:

- top 10 самых больших файлов;
- число silent catch;
- число `console.*`;
- покрытие критичных тестов;
- bundle size для frontend.

#### 4.6.3. Review policy

Ввести обязательные правила ревью:

- проверка на single responsibility;
- проверка на docs drift;
- проверка на file size budget;
- проверка на error handling.

#### 4.6.4. Definition of Done

Вынести в отдельный документ и начать применять ко всем PR.

### Результат недели

- проект не только стал чище, но и получил защиту от деградации;
- команда получает устойчивые правила разработки.

### Критерий готовности

- есть checklists;
- есть реальные quality gates;
- крупные проблемные узлы уже не монолитны.

## 5. Quick wins, которые можно сделать сразу

Это можно внедрить за 1–2 дня даже до полного roadmap:

- добавить `ESLint` и `Prettier`;
- запретить merge без lint;
- заменить пустые `catch` на явное поведение;
- ввести правило “никакого `console.*` в backend без причины”;
- перевести admin pages на lazy import;
- завести корневой инженерный стандарт в `docs/engineering/`;
- обновить `.env.example` по всем сервисам.

---

## 6. Что не делать во время этого roadmap

Чтобы не сломать темп и не утонуть в переписывании всего проекта, нельзя:

- одновременно переписывать все сервисы на TypeScript;
- параллельно менять архитектуру и продуктовую логику;
- делать глобальный rename всего репозитория без автоматизированных проверок;
- сначала строить «идеальную архитектуру», а потом думать о запуске.

Правильный подход:

- сначала дисциплина и границы;
- потом разрезание больших узлов;
- потом усиление тестов и типизации.

---

## 7. Риски roadmap

### Риск 1. Рефакторинг без safety net

Если сначала резать большие файлы без тестов и lint-инфраструктуры, появятся скрытые регрессии.

### Риск 2. Полумеры

Если оставить старые fake quality gates, процесс будет выглядеть серьёзно, но код останется незащищённым.

### Риск 3. Избыточный масштаб

Если пытаться переписать весь проект за раз, работа растянется и остановится.

### Риск 4. Несогласованные стандарты

Если утвердить документ, но не встроить его в PR checklist и CI, он быстро превратится в формальность.

---

## 8. Рекомендуемый порядок merge

Порядок PR:

1. tooling + configs + docs sync;
2. admin router split;
3. backend auth split;
4. miniapp router and telegram adapter cleanup;
5. bot validation and logging cleanup;
6. tests and guardrails hardening.

Такой порядок минимизирует риск и даёт быстрый эффект по читаемости.

---

## 9. Итог

Главная цель этого roadmap не в том, чтобы «сделать красиво».  
Главная цель — снизить стоимость изменений в проекте.

После выполнения плана FoodMiniApp должен стать:

- легче для входа новому разработчику;
- безопаснее для рефакторинга;
- понятнее при ревью;
- устойчивее к росту функциональности.

Если коротко, порядок действий такой:

- сначала правда и контроль качества;
- потом разрезание крупных файлов;
- потом выравнивание слоёв;
- потом закрепление правил через CI и ревью.
